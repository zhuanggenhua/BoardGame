# 审计失败分析：steampunk_aggromotive（蒸汽机车）

## 问题总结

**卡牌**：steampunk_aggromotive（蒸汽机车）  
**描述**：「打出到基地上。持续：如果你有一个随从在此基地，你在这里就拥有+5力量。」  
**当前实现**：使用 `'firstOwnerMinion'` 模式，只给拥有者在该基地的第一个随从加成 +5  
**缺陷**：第一个随从被移除（消灭/返回手牌/移动）后，+5 会"跳"到下一个随从身上

- 审计只验证了"给第一个己方随从 +5（避免重复计算）"，**没有质疑这个实现方式是否正确**

### 2. 审计维度的缺失

**当前审计维度（D1-D49）中缺少**：

- ❌ **D50：持续效果稳定性**（应该有但没有）
  - 持续效果的目标被移除/移动后，效果是否仍然正确？
  - 检查清单：

  - 检查清单：
    - [ ] 第一个元素被移除后，效果是否跳转到下一个？
    - [ ] 新元素插入到前面后，效果是否跳转？
    - [ ] 元素顺序改变后，效果是否仍然正确？

### 3. 实现假设的错误

**审计假设**：
- "给第一个随从 +5" ≈ "总战力 +5"
- 这个假设在**静态场景**下是正确的（第一个随从不变）
- 但在**动态场景**下是错误的（第一个随从会变）

**审计没有质疑**：
- 为什么要用"给第一个随从 +5"来模拟"总战力 +5"？
- 这种模拟方式在所有场景下都正确吗？
- 有没有更直接的实现方式？

## 审计流程的改进

### 1. 新增审计维度 D50

**D50：持续效果稳定性（强制）**

**触发条件**：新增/修改持续效果（ongoing/buff/debuff/光环）时触发

**核心原则**：持续效果的目标被移除/移动后，效果是否仍然正确？

**审查方法**：
1. **识别持续效果的目标类型**：
   - 目标是"具体实体"（如"第一个随从"）→ 必须测试目标移除/移动场景
   - 目标是"抽象条件"（如"所有己方随从"）→ 必须测试条件变化场景
   - 目标是"玩家/基地"（如"玩家总战力"）→ 通常不需要测试移除场景

2. **检查清单**（目标是具体实体时）：
   - [ ] 目标被消灭后，效果是否仍然正确？
   - [ ] 目标返回手牌后，效果是否仍然正确？
   - [ ] 目标移动到其他位置后，效果是否仍然正确？
   - [ ] 目标顺序改变后，效果是否仍然正确？

3. **判定标准**：
   - 效果应该跟随目标 → ✅ 目标移除后效果消失
   - 效果应该固定在位置 → ✅ 目标移除后效果转移到新目标
   - 效果应该独立于目标 → ✅ 目标移除后效果仍然存在
   - 效果行为不确定 → ❌ 实现有问题

4. **典型案例**：
   - ✅ "所有己方随从 +1 力量" — 目标是抽象条件，随从移除后自动不再受益
   - ❌ "第一个己方随从 +5 力量" — 目标是具体位置，随从移除后效果跳转（不符合"总战力 +5"语义）
   - ✅ "玩家在此基地 +5 总战力" — 目标是玩家，不依赖具体随从

## 总结

### 审计失败的根本原因

1. **审计维度不完整**：缺少 D50（持续效果稳定性）维度
2. **测试覆盖不足**：只测试了静态场景，没有测试动态场景
3. **实现假设错误**：假设"给第一个随从 +5" ≈ "总战力 +5"，但没有质疑这个假设
4. **代码审查不深入**：看到 `'firstOwnerMinion'` 模式时，没有追问"第一个随从被移除后会怎样？"

### 改进措施

1. **补充 D50 维度**：持续效果稳定性检查
2. **更新审计文档**：标注蒸汽机车的实现缺陷
3. **补充测试**：随从移除/移动场景测试
4. **代码审查清单**：添加"第一个/最后一个"逻辑的审查问题
5. **自动化检查**：创建 Property 测试，自动检查所有使用 `'firstOwnerMinion'` 的地方

### 教训

- **不要假设"模拟"等价于"真实"**：用"给第一个随从 +5"模拟"总战力 +5"看起来巧妙，但在动态场景下会出错
- **动态场景必须测试**：所有依赖"第一个/最后一个"的逻辑，必须测试顺序变化场景
- **审计要质疑实现方式**：不能只验证"实现了什么"，还要质疑"为什么这样实现"和"这样实现对吗"
