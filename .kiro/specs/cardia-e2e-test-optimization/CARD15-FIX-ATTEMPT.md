# Card15 (发明家) 修复尝试

## 尝试的修复

删除交互处理器（`registerInteractionHandler(ABILITY_IDS.INVENTOR, ...)`）

## 结果

❌ 修复失败

## 问题

第一次交互完成后，第二次交互弹窗没有显示。

## 根本原因

**能力执行器的两次交互模式没有被框架层正确支持**。

能力执行器的逻辑：
1. 第一次调用：`!ctx.selectedCardId && !ctx.selectedCards` → 创建第一次交互
2. 第二次调用：`ctx.selectedCardId && !ctx.selectedCards` → 创建第二次交互
3. 第三次调用：`ctx.selectedCards && ctx.selectedCards.length === 2` → 执行效果

**问题**：第一次交互完成后，框架层没有正确地：
- 设置 `ctx.selectedCardId` 为第一次选择的卡牌 ID
- 再次调用能力执行器以创建第二次交互

## 深层原因

查看代码发现，只有发明家使用了这种"两次独立交互"的模式。这说明：
1. 这个模式可能是后来添加的，框架层没有完全支持
2. 或者这个模式的实现不完整

## 正确的修复方案

### 方案 1：修改能力执行器为一次交互（推荐）

**原因**：让能力执行器创建一次交互，选择 2 张卡牌，然后恢复交互处理器。

**修改**：
1. 能力执行器：创建一次交互，`minSelect: 2, maxSelect: 2`
2. 交互处理器：恢复原来的实现（期望一次性接收 2 张卡牌）

**优点**：
- 符合框架层的设计（一次交互）
- 不需要修改框架层
- 交互处理器的实现是正确的

**缺点**：
- 用户体验可能不如两次交互清晰

### 方案 2：修复框架层支持两次交互

**原因**：让框架层正确支持"两次独立交互"模式。

**修改**：
1. 框架层：在第一次交互完成后，设置 `ctx.selectedCardId`，再次调用能力执行器
2. 能力执行器：保持不变

**优点**：
- 用户体验更好（两次交互更清晰）
- 支持更复杂的交互模式

**缺点**：
- 需要修改框架层
- 复杂度高

## 推荐方案

**方案 1：修改能力执行器为一次交互**

这是最简单、最直接的修复方案，符合框架层的设计。
