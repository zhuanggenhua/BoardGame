# Card09/10/14 E2E 测试调试计划

## 测试失败总结

### Card09 伏击者 ❌
- **错误**：手牌数量不对（期望 3 张，实际 4 张）
- **原因**：Academy 派系的牌没有被弃掉
- **能力**：选择一个派系，对手弃掉所有该派系的手牌
- **测试流程**：
  1. ✅ P1 打出伏击者（影响力 9）
  2. ✅ P2 打出傀儡师（影响力 10）
  3. ✅ 进入 ability 阶段
  4. ✅ 激活伏击者能力
  5. ✅ 派系选择弹窗显示
  6. ✅ 选择 Academy 派系
  7. ✅ 弹窗关闭
  8. ✅ 回合结束
  9. ❌ **P2 的 Academy 派系手牌没有被弃掉**

### Card10 傀儡师 ❌
- **错误**：卡在 ability 阶段，没有推进到 play 阶段
- **原因**：能力执行后没有正确结束
- **能力**：替换相对的牌（将对手的牌移到你的场上，你的牌移到对手的场上）
- **测试流程**：
  1. ✅ P1 打出傀儡师（影响力 10）
  2. ✅ P2 打出财务官（影响力 12）
  3. ✅ 进入 ability 阶段
  4. ❌ **卡在 ability 阶段，超时**

### Card14 女导师 ❌
- **错误**：复制发明家能力后，发明家的交互弹窗没有出现
- **原因**：复制的交互没有被正确创建或显示
- **能力**：复制并执行已打出牌的即时能力
- **测试流程**：
  1. ✅ P1 打出女导师（影响力 14）
  2. ✅ P2 打出精灵（影响力 16）
  3. ✅ 进入 ability 阶段
  4. ✅ 激活女导师能力
  5. ✅ 卡牌选择弹窗显示（选择要复制的能力）
  6. ✅ 选择发明家
  7. ✅ 确认选择
  8. ✅ 弹窗关闭
  9. ❌ **发明家能力的交互弹窗没有出现**

## 调试策略

### 1. Card09 伏击者
**假设**：交互处理器被调用了，但事件没有被正确应用

**验证步骤**：
1. 添加调试日志到 `registerFactionInteractionHandlers` 中的伏击者处理器
2. 检查 `CARDS_DISCARDED` 事件是否被发射
3. 检查 `reduceCardsDiscarded` 是否被调用
4. 检查弃牌逻辑是否正确

**可能的根因**：
- 交互处理器返回的 `state` 不会被应用（框架层 bug，与 Card15 相同）
- `CARDS_DISCARDED` 事件没有被正确 reduce

### 2. Card10 傀儡师
**假设**：能力没有被激活，或者激活后没有正确结束

**验证步骤**：
1. 检查傀儡师的能力实现
2. 添加调试日志
3. 检查是否有交互需要处理

**可能的根因**：
- 傀儡师能力需要交互，但测试没有处理
- 能力执行后没有正确推进阶段

### 3. Card14 女导师
**假设**：复制的交互没有被正确传递到 UI

**验证步骤**：
1. 检查女导师的能力实现中 `result.interaction` 的返回
2. 检查 `CardiaEventSystem` 是否正确处理复制的交互
3. 添加调试日志

**可能的根因**：
- 复制的交互的 `abilityId` 是被复制能力的 ID（如 `ability_i_inventor`），而不是女导师的 ID
- `CardiaEventSystem` 无法找到对应的交互处理器（因为 `sourceId` 是 `ability_i_inventor`，但注册的处理器是 `ability_i_governess`）

## 优先级

1. **Card14 女导师**（最可能是架构问题，需要修复复制交互的逻辑）
2. **Card09 伏击者**（可能是框架层 bug，与 Card15 相同）
3. **Card10 傀儡师**（需要先查看能力实现）

## 下一步

先调试 Card14 女导师，因为它最可能暴露架构问题。
