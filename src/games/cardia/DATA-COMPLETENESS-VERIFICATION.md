## 5. 图集索引映射

### 5.1 当前实现

- **数据层**：保留 `imageIndex: 1-48`（人类友好，1-based）
- **转换层**：新增 `getAtlasIndex(imageIndex)` 辅助函数
- **UI 层**：使用 `CardPreview` + `atlasIndex`

### 5.2 映射规则

```typescript
// 卡牌 1-10: imageIndex 1-10 → atlasIndex 0-9
if (imageIndex <= 10) {
    return imageIndex - 1;
}

// 卡牌 11-12: 缺失（返回 -1）
if (imageIndex === 11 || imageIndex === 12) {
    return -1;
}

// 卡牌 13-48: imageIndex 13-48 → atlasIndex 10-45
return imageIndex - 3;
```

### 5.3 缺失卡牌处理

- **卡牌 11**（大师工匠）：原始素材缺失，图集索引 10 为空白
- **卡牌 12**（将军）：原始素材缺失，图集索引 11 为空白
- **降级策略**：`getAtlasIndex()` 返回 -1 时，UI 显示派系颜色背景

## 6. 面向百游戏设计检查

根据 SKILL.md 1.5.4 节的"面向百游戏的设计检查"：

### 6.1 如果有 100 个游戏，每个游戏都这样做，会发生什么？

✅ **数据结构可扩展**：
- 新增游戏只需定义自己的 `CardDef` 接口
- 不会在框架层累积游戏特化代码
- 图集系统支持任意游戏复用

### 6.2 这个字段/逻辑是游戏特有的，还是可以抽象为通用能力？

✅ **合理分层**：
- `influence`：游戏特有（Cardia 的核心机制）
- `faction`：游戏特有（Cardia 的派系系统）
- `abilityIds`：通用模式（引用能力注册表）
- `imageIndex`：通用模式（图集索引）

### 6.3 这个实现是否依赖"已有游戏的具体实现"？

✅ **独立实现**：
- 使用 `engine/primitives/ability.ts` 通用能力框架
- 使用 `CardPreview` + 图集系统（SmashUp/SummonerWars 同款）
- 不依赖其他游戏的具体实现

### 6.4 如果规则变化，需要改多少地方？

✅ **单一数据源**：
- 修改卡牌属性：只改 `cardRegistry.ts`
- 修改能力效果：只改 `abilityRegistry.ts`
- 修改 UI 文案：只改 i18n JSON
