- 规则来源：`src/games/summonerwars/rule/召唤师战争规则.md`

- 现状：Summoner Wars 领域内核已实现，UI 需要重构为点击交互模式

## Goals / Non-Goals
- Goals:
  - 形成可运行的最小规则闭环（阶段、召唤/移动/建造/攻击/魔力/抽牌、胜利条件）。
  - 保持命令驱动与领域内核确定性；UI 只做交互表现。
  - 为后续完整卡牌与派系扩展预留数据驱动结构。
  - **实现类似游戏王的点击交互模式**：选中 → 确认，而非拖拽。
  - **手牌和棋盘使用同一套卡牌渲染组件**（精灵图切割）。
  - **注重游戏反馈**：召唤动画、攻击动画、伤害飘字等。
- Non-Goals:
  - 不覆盖完整卡牌库与所有事件/能力细节。
  - 不提供牌组构建 UI 或编辑器。

## Key Decisions
- **交互模式**：点击选中 → 点击目标，不使用拖拽
- **卡牌渲染**：使用精灵图集，通过 frameIndex 切割显示
- **动画反馈**：召唤飞行、移动滑动、攻击抖动、伤害飘字
- 使用 FlowSystem 管理阶段顺序与门禁，`sys.phase` 作为唯一阶段权威来源。
- 战场网格与初始部署使用配置驱动，避免硬编码棋盘尺寸与布局。
- 卡牌模型分为单位/事件/建筑三类，能力与复杂效果后续扩展。
- 魔力轨道范围 0-15，先手 2 / 后手 3 初始化；资源变更集中处理。

## Risks / Trade-offs
- 卡牌能力与事件效果未覆盖全部规则，需明确 MVP 范围，避免“规则不完整”导致误用。
- 若棋盘尺寸/布局未知，需提前确认或提供可配置默认值。

## Migration Plan
1. 在实现完成并通过测试前，保持 `manifest.enabled=false`。
2. 通过测试与最小 UI 验证后，启用 manifest 并更新清单。
