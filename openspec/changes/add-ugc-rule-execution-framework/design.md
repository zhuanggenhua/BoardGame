## Context
UGC 引擎主线定位为“需求 → 提示词 → 外部 AI → 粘贴导入”。系统仅提供规则执行框架与通用组件骨架，不内置任何游戏规则。当前能力已有 DomainCore 执行链，但缺少明确的框架规范，且与 `ugc-builder-v2` 的结构化效果块路线存在方向分叉风险。

## Goals / Non-Goals
- Goals:
  - 明确“规则执行框架 + 通用组件骨架”的能力边界
  - 约束规则来源为外部 AI 生成代码的粘贴导入
  - 确保预览与运行执行链一致
- Non-Goals:
  - 不提供任何游戏规则/牌型/比较逻辑
  - 不引入结构化效果块/DSL
  - 不提供手动代码编辑器

## Decisions
- 采用 DomainCore 契约作为唯一规则执行入口。
- 组件只提供动作钩子与渲染策略，不绑定规则。
- Builder 只保留“提示词 → 外部 AI → 粘贴导入”闭环。

## Alternatives Considered
- `ugc-builder-v2` 的分层模块化与结构化效果块路线：
  - 优点：更强结构化
  - 缺点：与当前“规则代码执行 + 外部 AI 粘贴导入”主线冲突
  - 结论：与本变更互斥，最终仅保留其一

## Risks / Trade-offs
- 规则质量完全依赖用户生成代码，需要更强的运行时校验与日志。
- 组件能力必须保持通用，避免逐步引入游戏特化逻辑。

## Migration Plan
- 若最终选择本变更为主线，则将 ugc-builder-v2 标记为弃用或留作备选。
