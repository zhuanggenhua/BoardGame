# 接入与验收清单

## 1. 审查输入

开始前至少阅读：

- `src/games/<gameId>/manifest.ts`
- `src/games/<gameId>/Board.tsx`
- `src/games/<gameId>/ui/`
- `docs/mobile-adaptation.md`

必要时补读：

- `src/components/common/MobileOrientationGuard.tsx`
- `src/components/common/MobileGestureWrapper.tsx`
- `src/hooks/ui/useMobileGestures.ts`

## 2. 风险分类

对每个风险点都标记归属：

- `framework`
  - 应沉到通用移动壳、公共 hook 或公共组件解决。
- `game`
  - 需要游戏层显式声明或小范围修正。
- `blocked`
  - 当前框架没有能力，必须先补框架层。

## 3. PC 护栏

- [ ] PC 桌面布局是否仍然保持原有列结构、主战区比例和视觉层级？
- [ ] 最近的移动端改动里，是否存在把桌面 `clamp(...)`、按钮尺寸、面板宽度全局改小的做法？
- [ ] 是否把桌面常驻结构错误地改成了所有视口都生效的抽屉 / 折叠？
- [ ] 任何尺寸压缩是否都被 `1023px` 移动视口或 shell 条件显式门控？
- [ ] 是否错误地把 `useCoarsePointer()` 当成“移动窄屏判断”来缩小桌面 UI？
- [ ] 是否存在 `src/index.css`、全局媒体查询或共享交互组件把多个区域的交互元素一起放大 / 缩小？
- [ ] 是否把触控命中区要求误写成了视觉尺寸要求，例如对整类交互元素统一加 `min-width` / `min-height`？
- [ ] 对于“变胖 / 变小 / 挤压”现象，是否已经定位到具体根因，而不是只记录了视觉结果？

## 4. 必查问题

- [ ] 有没有只能 hover 才能看到的重要信息？
- [ ] 有没有只能拖拽才能完成的关键操作？
- [ ] 有没有手机横屏下放不下的常驻侧栏？
- [ ] 有没有固定宽高或固定列数导致内容截断？
- [ ] 有没有关键按钮小于触控可用尺寸？
- [ ] 需要补触控可点性时，是否优先考虑显式 opt-in 命中区，而不是直接把交互元素视觉做胖？
- [ ] 有没有要求用户先缩放再点击的主路径？
- [ ] 有没有动画、粒子或阴影在移动端明显过重？
- [ ] 有没有只在桌面鼠标模型下才成立的交互假设？
- [ ] 当前异常究竟来自游戏组件本地 token，还是来自全局 CSS / shared button / 媒体查询污染？
- [ ] 能否指出导致异常的具体选择器、共享组件基类、媒体查询或条件判断？

## 5. 默认交互映射

- [ ] hover 详情默认映射为点击详情或长按预览。
- [ ] 关键拖拽默认补点击回退路径。
- [ ] 次要侧栏默认收进抽屉、标签页或切换面板，但只在移动条件下生效。
- [ ] 关键摘要信息默认保留在主视图，不把桌面核心信息整体折叠掉。
- [ ] 日志、规则说明、帮助默认降级为可切换面板。

如某项不适合默认映射，必须写明原因：

- 是玩法语义阻止了默认映射。
- 还是当前框架能力不足。
- 还是用户已有明确偏好与默认裁决冲突。

## 6. 游戏层最小改动标准

- [ ] manifest 明确声明移动支持级别。
- [ ] 主循环存在点击可达路径。
- [ ] 预览、状态说明、日志入口在触屏下可达。
- [ ] 折叠后的信息结构仍保留核心决策信息。
- [ ] 只有框架无法推断的部分才写游戏层例外。

## 7. 手机横屏验收

至少验证一个紧凑手机横屏和一个平板横屏。

建议基线：

- 手机横屏：约 `812x375`
- 大手机横屏：约 `932x430`
- 平板横屏：约 `1024x768`

验收动作：

- [ ] 进入对局。
- [ ] 查看手牌或主要资源。
- [ ] 查看状态、说明、日志。
- [ ] 完成一个完整主操作。
- [ ] 完成一次确认或取消。
- [ ] 在不缩放的情况下读懂当前局面。

## 8. PC 验收

至少对比一个常见桌面视口，例如 `1440x900` 或 `1920x1080`。

- [ ] 主战区、侧栏、顶部 / 底部操作区没有被移动端改动压缩。
- [ ] 桌面端 hover、按钮尺寸、面板宽度与改动前基线一致。
- [ ] 触控替代入口若在 PC 显示，不会挤压桌面核心布局。
- [ ] 用户原有的桌面主流程没有新增折叠、跳转或额外点击。

## 9. H5 视口 E2E 证据链

- [ ] 只为已声明移动支持或本次涉及移动适配的功能补移动视口 E2E。
- [ ] 优先参数化复用同一条测试流程，而不是复制两份测试文件。
- [ ] 每个支持移动的游戏通常只补 1 到 3 条关键移动验收路径。
- [ ] 至少覆盖 1 个手机横屏视口和 1 个平板横屏视口。
- [ ] 需要快速构造复杂局面时使用 TestHarness。
- [ ] 运行 `npm run test:e2e:ci -- <测试文件名>`。
- [ ] 查看并保留测试截图。
- [ ] 写 `evidence/<功能名>-e2e-test.md`，记录视口、交互路径和结果。

## 10. 容器升级门槛

只有下面都成立，才允许把游戏标成 `app-webview` 或 `mini-program-webview`：

- [ ] H5 / PWA 横屏适配已通过。
- [ ] 已有对应的 H5 视口 E2E 证据链。
- [ ] 主循环交互不再依赖 hover。
- [ ] 主循环交互不再依赖高精度拖拽。
- [ ] 移动端性能没有明显掉帧或卡顿。
- [ ] 登录、分享、拉起、业务域名等容器约束已明确。
