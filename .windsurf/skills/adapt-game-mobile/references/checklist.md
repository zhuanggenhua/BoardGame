# 接入与验收清单

## 1. 审计输入

开始前至少阅读：

- `src/games/<gameId>/manifest.ts`
- `src/games/<gameId>/Board.tsx`
- `src/games/<gameId>/ui/`
- `docs/mobile-adaptation.md`

必要时补读：

- `src/components/common/MobileOrientationGuard.tsx`
- `src/components/common/MobileGestureWrapper.tsx`
- `src/hooks/ui/useMobileGestures.ts`

## 2. 风险分类

对每个风险点都标记归属：

- `framework`
  - 应该沉到通用移动壳解决。
- `game`
  - 需要游戏层显式声明或小范围修正。
- `blocked`
  - 当前框架没有能力，必须先补框架层。

## 3. 必查问题

- [ ] 有没有只能 hover 才能看到的重要信息？
- [ ] 有没有只能拖拽才能完成的关键操作？
- [ ] 有没有手机横屏下放不下的常驻侧栏？
- [ ] 有没有固定宽高或固定列数导致内容截断？
- [ ] 有没有关键按钮小于触屏可用尺寸？
- [ ] 有没有要求用户先缩放再点击的主路径？
- [ ] 有没有动画、粒子或阴影在移动端明显过重？
- [ ] 有没有只在桌面鼠标模型下才成立的交互假设？

## 3.1 默认交互映射

审计时先套用这些默认映射，不要每次重做交互方案：

- [ ] hover 详情默认映射为点击详情或长按预览。
- [ ] 关键拖拽默认补点击回退路径。
- [ ] 次要侧栏默认收进抽屉、标签页或切换面板。
- [ ] 关键信息默认保留摘要在主视图，不完整常驻整块侧栏。
- [ ] 日志、规则说明、帮助默认降级为可切换面板。

如果某项不适合默认映射，必须明确写出原因：

- 是玩法语义阻止了默认映射
- 还是当前框架能力不足
- 还是用户已明确要求不同方案

## 4. 游戏层最小改动标准

做到这些就够，不要过度重写：

- [ ] manifest 明确声明移动支持级别。
- [ ] 主循环存在点击可达路径。
- [ ] 预览、状态说明、日志入口在触屏下可达。
- [ ] 折叠后的信息架构仍保留核心决策信息。
- [ ] 只有框架无法推断的部分才写游戏层例外。

## 4.1 何时需要追问用户

默认不追问：

- hover 的基础替代方式
- 次要侧栏的折叠方式
- 是否补点击回退路径

只有这些情况才追问：

- [ ] 方案差异会改变玩法语义
- [ ] 规则文档无法判断信息优先级
- [ ] 用户已有明确偏好与默认裁决冲突

## 5. 手机横屏验收

至少验证一个紧凑手机横屏和一个平板横屏。

建议基线：

- 手机横屏：约 `812x375`
- 大手机横屏：约 `932x430`
- 平板横屏：约 `1024x768`

验收动作：

- [ ] 进入对局
- [ ] 查看手牌或主要资源
- [ ] 查看状态、说明、日志
- [ ] 完成一个完整主操作
- [ ] 完成一次确认或取消
- [ ] 在不缩放的情况下读懂当前局面

## 6. 竖屏兜底

竖屏不是默认完整目标，但至少要明确行为：

- [ ] 给出清晰的横屏引导，或明确降级说明。
- [ ] 不把竖屏误报成“完整支持”。
- [ ] 不要求用户通过不断缩放来代替交互设计。

## 7. 容器升级门槛

只有下面都成立，才允许把游戏标成 `app-webview` 或 `mini-program-webview`：

- [ ] H5 / PWA 横屏适配已通过。
- [ ] 主循环交互不再依赖 hover。
- [ ] 主循环交互不再依赖高精度拖拽。
- [ ] 移动端性能没有明显掉帧或卡顿。
- [ ] 登录、分享、拉起、业务域名等容器约束已明确。
